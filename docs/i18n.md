# Translations (i18n)

**react-intl** + **FormatJS** for extracting and syncing frontend strings.

Doc index: [README](../README.md)

See also: [Team SOP — Before you push](./team-sop.md#before-you-push) (`pnpm i18n:extract`) · [Local Development](./local-development.md)

## Writing translatable content

All user-facing text in the application must be written in a way that can be extracted by the i18n system. Instead of writing plain text directly inside JSX, every translatable string should be wrapped using either `<FormattedMessage />` or `intl.formatMessage()`.

For example, a simple static text like:

```jsx
<div>Hello world</div>
```

must be rewritten into a translatable format:

```jsx
<div>
    <FormattedMessage defaultMessage="Hello world" />
</div>
```

Alternatively, when translation is needed inside logic or variables, the `intl.formatMessage` function can be used:

```js
intl.formatMessage({
    defaultMessage: 'Hello world',
});
```

The key idea is that **every human-readable string must pass through the i18n system**, so it can be extracted and managed automatically.

## Extraction Process

Once translatable messages are added to the codebase, the extraction process is used to collect all `defaultMessage` entries from the React source code and generate a centralized English translation file.

This is done using the command:

```bash
pnpm i18n:extract
```

When executed, the FormatJS extractor scans all React source files and builds or updates the file:

```
resources/js/lang/en.json
```

This file becomes the **source of truth for all translation keys in the application**. It contains every message detected in the codebase, based on the `defaultMessage` values provided in components and functions.

At this stage, the system does not yet handle other languages. It only ensures that all messages used in the UI are collected in a single place.

## Generating Translation Differences

After the English source file (`en.json`) is generated, the next step is to prepare translation updates for a specific language.

This is done using:

```bash
pnpm i18n:diff id
```

This command compares:

- `en.json` (source language)
- `id.json` (target language, if it exists)

The system then identifies all keys that exist in `en.json` but are missing in `id.json`.

These missing entries are exported into:

```
resources/js/lang/id.diff.json
```

Each entry in the diff file uses the English text as the default value. This ensures that translators always have readable reference text while translating.

If `id.json` does not exist yet, the system treats it as empty and generates a full set of translatable entries.

## Translating Content

Translation work is done entirely inside the generated diff file:

```
id.diff.json
```

Each key represents a UI message, and the value contains the English default text. Translators are expected to replace these values with the appropriate translation for the target language.

This workflow ensures that translation changes are isolated and do not interfere with existing stable translations stored in `id.json`.

## Merging Translations

Once translation is complete inside `id.diff.json`, the final step is to merge the translated content into the main language file.

This is done using:

```bash
pnpm i18n:merge id
```

The merge process will take all entries from `id.diff.json` and insert them into `id.json`, but only if the keys do not already exist. Existing translations are preserved and never overwritten, ensuring that previously approved translations remain stable.

After a successful merge, the diff file is automatically removed, and `id.json` becomes the updated translation source for the application.

## Summary of Workflow

The overall workflow is designed to be predictable and safe for teams working on translations. Developers only write `defaultMessage` in React components, while the system handles extraction, comparison, and merging automatically. Translation work is isolated into diff files, ensuring that translators only focus on missing or new content without modifying existing approved translations.

This approach keeps the translation system fully synchronized with the codebase while minimizing manual maintenance overhead.
